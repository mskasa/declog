# 0010: kizami へのリネームとスコープの拡張（リビングドキュメント対応）

- Date: 2026-03-17
- Status: Active
- Author: masahiro.kasatani
- Supersedes: 0006

## Context

declog はもともと ADR（アーキテクチャ決定記録）専用ツールとして設計された。
コアバリューは「決定の背景にある理由を記録する」ことだった。

しかし、ソフトウェア開発チームが抱える本質的な問題はより広い。
**設計書（アーキテクチャ図、API仕様書、詳細設計書）が、実際のコードと
常に乖離し続けるという問題である。** 著者がこれまで関わったすべての現場で、
設計ドキュメントはコードベースと同期していなかった——100%の確率で。

`why audit` は、ドキュメントの `## Related Files` セクションに記載されたファイルが
リポジトリに存在するかを検証することでこの乖離を検出する仕組みを導入した。
この仕組みは ADR に限定されない——ソースファイルを参照する任意の Markdown ドキュメントに適用できる。

ここから気づきが得られた。このツールのコアバリューは「ADR管理」ではなく、
**「Git ネイティブなリビングドキュメントの乖離検出」** である。ADR はリビングドキュメントの
一種に過ぎず、同じアプローチが設計書・API仕様書・アーキテクチャ図の参照管理にも使える。

既存のツール名（`declog` / `why`）は「決定ログ」を意味しており、拡張後のスコープには
狭すぎる。新しいユーザーからは ADR 専用ツールに見えてしまう。

## Decision

1. **ツール名を `kizami`（刻み）に変更する。**
   設計上の決定やアーキテクチャをコードベースに永続的に刻み込むという
   コンセプトを表している。コマンド名も `kizami`（`why` を置き換え）とする。

2. **スコープをリビングドキュメント全般に拡張する。**
   ADR だけでなく、`## Related Files` セクションを持つ任意の Markdown ドキュメントを
   対象とする。設計書・API仕様書など新しいドキュメント種別は `kizami log --type` フラグで対応する。

3. **`kizami audit` を汎用化する。**
   `docs/decisions/` 固定ではなく、複数の設定可能なディレクトリをスキャンできるようにする。

`## Related Files` の仕組みと Git ネイティブの哲学はそのまま継承する。

## Consequences

- インターネット上で一意の、検索しやすい名前を持つツールになる
- 拡張されたスコープは、ほぼすべてのソフトウェアプロジェクトで感じられる
  痛みを解決する
- ADR 機能は完全に保持される。既存ユーザーへの移行パスも存在する
- `kizami audit` がコアの差別化機能となる：テキストエディタと Git 以外の
  インフラを必要としない、軽量で Git ネイティブな乖離検出
- コマンド名変更（`why` → `kizami`）は破壊的変更だが、パブリックリリース前のため影響は最小限
- 日本語由来の名前により、国際的な OSS エコシステムでの独自性が生まれる

## Alternatives Considered

- **`declog` / `why` のままスコープを拡張する：** 名前が拡張後の目的と矛盾し、
  設計書管理を求める新規ユーザーを混乱させる
- **設計書向けに別ツールを作成する：** コア監査機能が重複し、メンテナンスコストが増大する
- **`docrift` や `driftless` に改名する：** 説明的だが `kizami` ほど一意でなく印象に残りにくい
