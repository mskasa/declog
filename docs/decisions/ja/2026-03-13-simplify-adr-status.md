# ADRステータスをActive・Inactive・Supersededの3つに絞る

- Date: 2026-03-13
- Status: Active
- Author: masahiro.kasatani

## Context

当初の4つのステータス（`Proposed`・`Accepted`・`Superseded`・`Deprecated`）は、意思決定がレビューと承認プロセスを経るワークフロー向けのADRツールから引き継いだものだった。

declogの運用ではADRをコード変更と同時にコミットする。ADRを書いた時点でその判断はすでに確定しているため、`Proposed` と `Accepted` は形骸化したステータスになる。また `Deprecated` は「単純に無効になった」ケースと「新しいADRに置き換えられた」ケースを区別できず、設計の変遷を追いにくい。

以下の2つの無効化シナリオを区別できる必要がある：
1. 新しいADRに置き換えられた（置き換え先へのトレーサビリティが重要）
2. 置き換えなしで無効になった（例：機能を丸ごと削除した場合）

## Decision

ステータスを以下の3つに絞る：

| ステータス           | 意味                             |
| -------------------- | -------------------------------- |
| `Active`             | 現在有効な判断（デフォルト）     |
| `Inactive`           | 無効。置き換え先なし             |
| `Superseded by NNNN` | ADR NNNNに置き換えられた         |

デフォルトは `Active`。コード変更と同時にコミットする運用のため、作成時点で意思決定済みとみなす。

## Consequences

- ステータスフィールドが常に意味を持つ。「形式的なステータス」が存在しない
- `Superseded by NNNN` に置き換え先のADR番号が直接埋め込まれるため、設計の変遷が別途参照なしに追跡できる
- メンタルモデルがシンプルになる。各ステータスが明確な意味を持つ
- 既存のADR（0001〜0007）は `Accepted` を使用しているが、これは `Active` と同等とみなし、随時更新していく

## Alternatives Considered

- **元の4ステータスを維持：** MADRツールとの互換性はあるが、ADRをコードと同時にコミットする運用では実質使われないステータスが生まれる
- **2ステータス（Active / Inactive）：** よりシンプルだが、どのADRが何を置き換えたかのトレーサビリティが失われる
- **自由記述のステータス：** 柔軟性は最大だが、ADR間でセマンティクスが統一されない
