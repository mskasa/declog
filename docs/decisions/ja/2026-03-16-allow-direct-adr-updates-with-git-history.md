# GitHistoryを活用したADRの直接更新を許容する

- Date: 2026-03-16
- Status: Active
- Author: masahiro.kasatani

## Context

ADRのイミュータブル原則は、分散バージョン管理システムが登場する以前の時代に生まれたものである。Git以前の環境ではドキュメントを上書きすると履歴が失われるため、イミュータブルにすることが唯一の安全策だった。

Supersededパターンを厳格に運用すると、判断内容を更新するたびに元のADRをコピー・修正した新しいADRを作成する必要がある。これにより、旧ADRと新ADRの差分を理解するには2つのファイルを手動で比較するしかなく、`git diff` の恩恵を受けられない。

Gitが存在する環境では、すべてのファイルの変更が作者・タイムスタンプ・コミットメッセージとともに記録される。`git log -- path/to/adr.md` でADRの完全な変更履歴をいつでも参照できる。履歴保全のために明瞭さを犠牲にする技術的な必要性はない。

## Decision

ADRはGitの履歴管理を前提に直接更新を許容する。何が変わったか・なぜ変えたかの根拠はGit履歴が担う。

運用ガイドライン：
- 同じ判断を修正・精緻化する場合はADRをそのまま更新する
- `git diff` で変更内容を伝え、コミットメッセージで変更理由を説明する
- ADR更新時のコミットメッセージには何をなぜ変えたかを明記する（例：`docs: update ADR 0003 - increase pool size from 10 to 20 based on load test`）
- 判断の方向性ごと変わった場合（根本的に異なる選択をした場合）は新しいADRを作成し、旧ADRを `Superseded by NNNN` にする

「同じ判断の修正」と「方向性が変わった新判断」の境界はケースバイケースの判断になる。迷った場合は新しいADRを作成し、変遷を明確に残す方を選ぶ。

## Consequences

- 重複ファイルが増えることなくADRの内容を最新の状態に保てる
- `git diff` により更新内容を正確にレビュー可能な変更記録として残せる
- 細かな修正のたびに新しいADRを作成するオーバーヘッドなく判断を継続的に改善できる
- Supersededステータスは本質的な方向転換に対してのみ使われるため、設計の変遷を引き続き追跡できる
- コミットメッセージの重要性が高まる — 「update adr」のような曖昧なメッセージは不十分

## Alternatives Considered

- **厳格なイミュータブル運用（元の方針）：** シンプルで安全だが、ファイルが増殖し、判断の段階的な進化を追うのが難しくなる
- **追記のみ（末尾に変更履歴セクションを追加）：** 元の内容を残せるが、ノイズが増え、他のソースファイルの管理方法と一貫性がない
