# ドキュメントから数値 ID を廃止する

- Date: 2026-03-23
- Status: Active
- Author: masahiro.kasatani

## Context

kizami が作成するドキュメントは現在、ファイル名に 4 桁の数値 ID プレフィックスを使用しています（例: `0001-use-go-over-shell-script.md`）。この ID はファイル名、ドキュメント見出し（`# 0001: Title`）、および廃止メタデータ（`- Supersedes: 0003`、`Status: Superseded by 0007`）に埋め込まれています。

このスキームは CLI での短縮参照（`kizami show 3`）と廃止関係をコンパクトに表現するために採用されました。

しかし、ID ベースのアプローチには構造的な制約があります。

- **サブディレクトリ整理が機能しない**: `NextID` は直下のディレクトリのみをスキャンします。ドキュメントをサブディレクトリ（例: `docs/decisions/2024/`、`docs/decisions/ja/`）に配置すると、ID の衝突やファイルの見落としが発生します。
- **複雑なロジック**: `NextID`、`FindByID`、`List` はすべて 4 桁フォーマット固有の処理を含んでいます。
- **MADR 規約との非互換**: 広く使われている ADR エコシステム（adr-tools、MADR）は数値 ID なしの日付・スラッグベースのファイル名を使用しています。
- **不透明な参照**: `Superseded by 0007` はファイルを参照しないと意味がわかりません。ファイル名スラッグは自己文書化されています。

## Decision

ドキュメントのファイル名、見出し、メタデータから数値 ID を廃止します。

**新しいファイル名形式**: `YYYY-MM-DD-kebab-case-title.md`
**新しい見出し形式**: `# Title`（ID プレフィックスなし）
**廃止メタデータ**: `- Supersedes: <slug>`（拡張子なしのファイル名）

日付プレフィックスにより時系列ソートは維持されます。スラッグは CLI コマンドや相互参照で使用できる安定した人間可読な識別子です。

現在数値 ID を受け取る CLI コマンドはスラッグを受け取るよう変更します。

- `kizami show <slug>`
- `kizami status <slug> <status>`
- `kizami supersede <slug> <slug>`

## Consequences

**メリット:**
- ID の衝突なしにサブディレクトリ整理（年別、言語別、チーム別など）が可能になる
- `NextID`、`FindByID` および関連する採番ロジックを削除でき、コードベースが簡略化される
- ファイル名が自己文書化される。`0007` が何かを調べる必要がなくなる
- MADR・adr-tools の規約に近づく

**トレードオフ:**
- 破壊的変更: `NNNN-` プレフィックスを持つ既存ドキュメントの移行が必要（ファイルリネーム、見出しと相互参照の更新）
- `kizami show 3` の短縮記法が失われる。スラッグを使う必要がある
- スラッグベースの検索は整数比較ではなくプレフィックスマッチや完全一致が必要になる

**移行パス:**
- 移行期間中、パーサーは `NNNN-slug.md` と `YYYY-MM-DD-slug.md` の両形式を受け付け、既存リポジトリとの互換性を保つ

## Alternatives Considered

**ID を維持し、グローバル ID レジストリでサブディレクトリをサポートする**
サブディレクトリ間で ID を調整するためのロックファイルやインデックスが必要になります。根本原因を解決せずに複雑さが増すだけです。

**ID を維持し、ドキュメントをフラットなディレクトリに制限する**
言語別ディレクトリや年別アーカイブなど正当な整理ニーズを妨げます。増加するドキュメントセットを扱う他ツールとの扱い方が一致しません。

**連番 ID の代わりに UUID を使用する**
衝突はなくなりますが完全に不透明になり、コミットメッセージや PR 説明での人間による参照と非互換です。
