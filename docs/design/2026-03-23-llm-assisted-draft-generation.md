# LLM-Assisted Draft Generation

- Date: 2026-03-23
- Type: Design
- Status: Active
- Author: masahiro.kasatani

## Overview

`kizami adr --ai` and `kizami design --ai` use the Anthropic API to generate draft document sections from the current git context (changed files and staged diff), reducing the friction of writing living documents alongside code changes.

## Background

Writing useful ADRs and design documents requires capturing context that is already implicit in the code change: what files were touched, what the diff shows, and what problem was being solved. Without assistance, this context must be reconstructed from memory after the fact, which degrades quality and discourages the habit.

By feeding the staged diff and changed file list directly to an LLM at document creation time, kizami can produce a meaningful first draft that the developer then edits to completion — rather than starting from an empty template.

## Goals / Non-Goals

**Goals:**
- Generate body sections (Context/Decision/Consequences or Overview/Background/Design etc.) from git context
- Keep the standard metadata header (Date, Type, Status, Author, Supersedes) deterministic — never generated by the LLM
- Support a `--dry-run` flag to preview the prompt without calling the API
- Support a `--model` flag and `[ai] model` config key to override the default model
- Work for both ADR (`kizami adr --ai`) and design document (`kizami design --ai`) creation flows

**Non-Goals:**
- Streaming output — the full response is awaited before writing the file
- Automatic submission without human review — the editor is always opened after generation
- Generating metadata fields (date, status, author) — these are always hardcoded
- Supporting LLM providers other than Anthropic

## Design

### Flow

```
kizami adr --ai "<title>"
  │
  ├── 1. promptSimilar()         — same as non-AI flow; check for similar docs
  ├── 2. GatherInput()           — collect changed files + staged diff from git
  ├── 3. BuildPrompt()           — construct prompt string
  ├── 4. [--dry-run] DryRun()    — show prompt, ask for confirmation
  ├── 5. GenerateDraft()         — call Anthropic Messages API
  ├── 6. RenderHeader()          — build deterministic metadata header
  ├── 7. CreateFromDraft()       — write header + LLM body to YYYY-MM-DD-slug.md
  └── 8. openEditor()            — open file in editor for review
```

### Context Gathering (`internal/ai/prompt.go`)

`GatherInput(dir, title string)` collects:
- **Changed files**: staged files from `git diff --staged --name-only`
- **Staged diff**: output of `git diff --staged`, truncated to `DiffLimit = 2000` characters to stay within token budgets

Both are limited to staged files so that the developer controls the context by staging before running `kizami adr --ai`.

The diff is truncated rather than summarized to keep the implementation simple and deterministic. The 2000-character limit was chosen as a practical balance between context quality and API cost.

### Prompt Construction

Two separate prompt builders exist for the two document types:

- `BuildPrompt(input)` — instructs the LLM to generate `## Context`, `## Decision`, `## Consequences`, and `## Related Files` sections in Markdown
- `BuildDesignPrompt(input)` — instructs the LLM to generate `## Overview`, `## Background`, `## Goals / Non-Goals`, `## Design`, `## Implementation Plan`, `## Open Questions`, and `## Related Files` sections

Both prompts:
- Specify "Output the Markdown only. No explanation or preamble." to suppress prose wrapping
- List the changed files and include the truncated diff as context
- Do not ask the LLM to produce the metadata header (date, status, author)

### Header / Body Split

The metadata header is always rendered deterministically by `RenderHeader` / `RenderDesignHeader` in `internal/template/template.go`. The LLM only generates the body sections starting from `## Context` or `## Overview`.

`CreateFromDraft` and `CreateDesignFromDraft` in `internal/decision/generate.go` concatenate:

```
header  =  RenderHeader(title, author, supersededSlug)
content =  header + "\n" + draft
```

This ensures that Date, Status (`Draft`), Author, and Supersedes are never hallucinated.

### API Call (`internal/ai/draft.go`)

`GenerateDraft(prompt, model, apiKey string)` calls `POST https://api.anthropic.com/v1/messages` with:
- `model`: resolved via `ResolveModel(flagModel, cfg)` — flag > config > `DefaultModel`
- `max_tokens`: 2048
- A single `user` message containing the prompt

The `ANTHROPIC_API_KEY` environment variable is required. If absent, an actionable error message is returned before any network call is made.

### Model Resolution

Priority (highest to lowest):
1. `--model` flag on the command line
2. `[ai] model` in `kizami.toml` or `~/.config/kizami/config.toml`
3. Hard-coded default: `claude-sonnet-4-20250514` (`config.DefaultModel`)

### `--dry-run` Flag

`DryRun(prompt, r, w)` prints the full prompt to stdout and prompts the user for `y/n` confirmation before the API is called. This is useful for inspecting prompt quality or estimating token cost without incurring charges.

## Open Questions

- **Diff truncation**: The 2000-character hard cut may split a diff mid-hunk, potentially confusing the LLM. A smarter truncation (e.g., whole hunks only, or file-by-file priority) could improve draft quality for large changesets.
- **Unstaged changes**: Both the changed file list and the diff are limited to staged files. This is intentional — the developer controls what context is sent by staging files before running `kizami adr --ai`. Unstaged changes are excluded to avoid including unrelated work-in-progress edits.
- **Token usage feedback**: The API response includes token counts, but kizami does not currently surface them. Logging this at `--dry-run` time would help developers tune their workflow.
