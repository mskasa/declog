# Scheduled CI Workflows

- Date: 2026-03-23
- Type: Design
- Status: Active
- Author: masahiro.kasatani

## Overview

kizami generates two scheduled GitHub Actions workflows: `adr-audit.yml` runs `kizami audit` weekly and opens a GitHub Issue when stale references are found; `kizami-promote.yml` automatically promotes documents from `Status: Draft` to `Status: Active` on every push to `main`.

## Background

Living documents require two ongoing maintenance signals: detecting when documents have drifted from the code they describe, and ensuring that draft documents written during development are formally activated once the work is merged.

Both of these are periodic tasks that no individual developer should have to remember. Encoding them as scheduled or event-driven CI workflows ensures they run consistently regardless of team habits.

## Goals / Non-Goals

**Goals:**
- `adr-audit.yml`: run `kizami audit` on a weekly schedule and on manual trigger; create a GitHub Issue when stale references are found; deduplicate issues to avoid spam
- `kizami-promote.yml`: promote all `Status: Draft` documents to `Status: Active` on push to `main`; commit the changes as a bot commit with `[skip ci]` to avoid triggering CI loops
- Both workflows are generated by `kizami init` (opt-in)
- Both workflows are self-contained — they install kizami from the published release and require no other tooling

**Non-Goals:**
- Automatically fixing stale references (audit reports only; humans decide the fix)
- Promoting documents on branches other than `main`
- Supporting CI systems other than GitHub Actions

## Design

### Audit Workflow (`adr-audit.yml`)

#### Trigger

- `schedule: cron: '0 0 * * 1'` — runs at 00:00 UTC every Monday
- `workflow_dispatch` — allows manual runs from the GitHub Actions UI

#### Steps

1. **Checkout** with `fetch-depth: 0` — full history is required for `kizami audit` to walk the full document set
2. **Install Go** (`actions/setup-go`)
3. **Install kizami** via `go install github.com/mskasa/kizami@latest` — always uses the published release, keeping the workflow independent of the local Go toolchain
4. **Run `kizami audit`** with step id `audit` — the command sets the output `stale_found=true` if any stale references are detected
5. **Create GitHub Issue** (conditional on `steps.audit.outputs.stale_found == 'true'`) using `actions/github-script`

#### Issue Deduplication

Before creating an issue, the script lists all open issues and checks whether any title contains `[kizami audit]`. If a duplicate exists, no new issue is created. This prevents a backlog of identical weekly issues when the stale references are not promptly fixed.

The issue title includes the current date:
```
[kizami audit] Stale file references detected (2026-03-23)
```

The issue body contains the full `kizami audit` output via the `AUDIT_RESULT` environment variable.

#### Required Permissions

The job needs `issues: write` to create GitHub Issues. This is declared explicitly in the job's `permissions` block.

### Promote Workflow (`kizami-promote.yml`)

#### Trigger

- `push` to `main` — runs on every merge to the default branch

#### Rationale for Draft → Active on Merge

Documents are created with `Status: Draft` during development (while a decision is still in progress or being reviewed). When the feature branch is merged to `main`, the decision is considered finalized. Automatically promoting at merge time aligns the document status with the code lifecycle without requiring a manual status update.

#### Steps

1. **Checkout** — default shallow checkout is sufficient (no git history needed)
2. **Promote Draft documents** — a shell script scans `docs/decisions` and `docs/design` for `.md` files containing `- Status: Draft`, replaces with `- Status: Active` using `sed`, and sets `changed=1` if any file was modified
3. **Commit** (conditional on `changed == 1`) — commits with message `docs: auto-promote Draft documents to Active [skip ci]` as `github-actions[bot]`; `[skip ci]` prevents the push from re-triggering this workflow

#### Customization Points

The workflow is intentionally self-documented with inline comments pointing to the two customization points:
- The `main` branch name (for repos using `master` or a different default branch)
- The `dirs` variable listing directories to scan

These are left as editable fields in the generated file rather than wired to `kizami.toml`, since the workflow runs in GitHub Actions without access to the local config file.

#### Bot Commit Identity

The commit uses:
```
user.name:  github-actions[bot]
user.email: github-actions[bot]@users.noreply.github.com
```

This is the standard GitHub Actions bot identity, which produces a recognizable commit in the git log and avoids attribution to any individual developer.

## Open Questions

- **`kizami audit` exit code**: Currently `kizami audit` does not set a non-zero exit code when stale references are found (to avoid breaking local shells). The CI workflow relies on a GitHub Actions output variable (`stale_found`) instead. A future `--strict` flag could make the command exit non-zero for CI use.
- **Promote on non-main branches**: Some teams use a `release/` branch model. The promote workflow only targets `main` and must be manually edited for other patterns.
- **Promote scope**: The workflow promotes all Draft documents across all scanned directories. There is no per-document opt-out. A `promote: never` metadata field could exempt specific documents from auto-promotion.
